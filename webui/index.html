How this RAG works

A hybrid-retrieval RAG over your Obsidian vault. Its core is two search lanes — Dense (semantic / vectors) and Sparse (keyword / BM25) — run together and fused with Reciprocal Rank Fusion; optional branches can widen or reshape the query and add extra lanes. The fused pool is re-ranked by a cross-encoder, then a cited answer is grounded in your own materials. Everything up to generation is local; only the final answer needs the LLM.

Solid = always-on; dashed = optional (toggle, preset, or automatic); blue = the automatic scope router; dotted = the untouched original question. Every retrieval lane — the two core lanes, scope-filtered lanes, HyPE / Code / Live-vault — lands in the same RRF pool, so adding a lane can only add candidates, never hide any. HyDE reshapes what retrieval searches for, but the cross-encoder scores — and the generator answers — your original question. dense_top_k / sparse_top_k size the candidate pool, rerank_top_k decides how many chunks reach the LLM, max_tokens caps the answer.
The whole ingest side has its own diagram — flip the toggle above.
Solid = always-on; dashed = optional flags; blue = automatic. Four loaders, one JSONL contract: whatever the format, every chunk lands in data\*_chunks.jsonl with the same doc_id rule and metadata — that is the corpus. The indexes are derived: lose either and it rebuilds from the JSONLs. index --append embeds only new doc_ids, so re-running anything is safe; text edits mint new doc_ids (swap playbook), metadata edits go through retag with no re-embed. The chunking picker (heading | fixed) only changes how oversized sections split — small sections are identical either way. The query API keeps indexes warm — restart :8051 after any index change.

The at-a-glance dashboard: total chunks & documents, ChromaDB vector count, disk usage, the per-domain chunk share (the “shelf”), which JSONL files hold what, and the most recent jobs. Read-only.

Documents

Vault

Browse the real vault tree with ✓ in-RAG badges + chunk counts, stage domain/tag edits on indexed files and Apply as one batch, and search filenames across the whole vault (including books outside the browse root). Never writes to your vault.

Jobs

The serial job queue and live logs. Tail any job’s output, cancel a running one, and retry a failed one — every job is idempotent, so a retry never duplicates committed work. A zero-result ingest fails (not a silent “done”).

The Query knobs — what each one changes

Everything is per-call and leaves the warm defaults untouched. The retrieval echo under each answer shows what actually ran. General rule: tune retrieval on Search only (free, local, shows scores) and spend one Ask once the right chunks are surfacing.

The fusion math (RRF)

Each lane returns its own ranked list. A document at rank r in a lane contributes \( 1/(k+r) \); a doc’s fused score is the sum across every lane it appears in. Then the cross-encoder reranks the top of that pool.

$$ \text{score}(d) \;=\; \sum_{\ell \,\in\, \text{lanes}} \frac{1}{k + r_\ell(d)}, \qquad k = 60 $$

Why \(k=60\)? It’s the smoothing constant from the original RRF paper (Cormack et al., 2009) and the de-facto default. With \(k=60\), rank 0 ⟶ \(1/60\!\approx\!0.0167\) and rank 1 ⟶ \(1/61\!\approx\!0.0164\): the first ~dozen ranks contribute almost equally, so no single lane’s exact ordering dominates. A small k sharpens (the #1 hit in each lane wins); a large k flattens (rank barely matters). 60 is the tested middle ground — kept as-is.

Why no per-lane weights? Weighted RRF (\( \sum w_\ell/(k+r_\ell) \)) is a legitimate but marginal tweak here: a cross-encoder reranker sits downstream and re-scores the fused pool, so as long as the right chunks are in the pool (governed by dense_top_k / sparse_top_k), the reranker decides the final order. Weights would mostly re-add a tuning burden for gains within noise — so the honest levers are the per-lane pool sizes + rerank_top_k above, not fusion weights.
